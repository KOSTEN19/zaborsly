# Zaborsky ANPR

Система распознавания российских госномеров с двух RTSP-камер, логированием заездов/выездов и веб-админкой.

> **Docker Desktop на Windows?** Используйте версию в папке [`zaborsky-windows`](../zaborsky-windows) — порт 8080, лимиты RAM, настройки под WSL2.

## Возможности

- Чтение RTSP с 2 камер
- Распознавание российских номеров (nomeroff_net)
- Автоопределение въезда/выезда по очерёдности срабатывания камер
- Сохранение фото и журнала в PostgreSQL
- Веб-админка: дашборд, **live-просмотр камер**, журнал сессий, сырые детекции, настройки

## Требования

- Linux-сервер с Docker и Docker Compose v2
- Минимум 4 CPU / 8 GB RAM (CPU-only inference)
- Камеры доступны по RTSP из сети сервера

## Быстрый старт

```bash
cp .env.example .env
# Отредактируйте .env: пароли, RTSP-URL камер

docker compose up -d --build
```

Откройте `http://<IP-сервера>/` — логин и пароль из `.env` (`ADMIN_USERNAME` / `ADMIN_PASSWORD`).

## Настройка камеры

В `.env` укажите **одну** камеру:

```env
CAMERA_1_RTSP=rtsp://user:pass@192.168.1.101:554/stream1
CAMERA_1_NAME=Камера шлагбаум
```

Вторая камера **опциональна** — добавьте `CAMERA_2_RTSP`, и система переключится в режим 2 камер.

### Тест без камеры

Для отладки можно использовать локальные видеофайлы вместо RTSP:

```env
VIDEO_FILE_1=/data/test/cam1.mp4
VIDEO_FILE_2=/data/test/cam2.mp4
```

Без настроенных камер worker будет ждать и писать в лог, что нужно задать RTSP или видеофайлы.

## Логика въезд / выезд

### 1 камера (по умолчанию)

- Первое распознавание номера → **въезд** (сессия `on_site`)
- Повторное распознавание того же номера (после cooldown) → **выезд** (сессия `completed`)
- Подходит для одного проезда через шлагбаум

### 2 камеры (если задан `CAMERA_2_RTSP`)

1. Номер пойман на камере 1 в момент T1
2. Тот же номер на камере 2 в T2 (в пределах `MOVEMENT_WINDOW_SEC`)
3. Пара Cam1→Cam2 = направление из `CAM1_TO_CAM2_DIRECTION`
4. Пара Cam2→Cam1 = противоположное направление

## Оптимизация (качество + скорость)

Два независимых пайплайна:

| Пайплайн | Назначение | Настройки |
|----------|------------|-----------|
| Live | Плавный просмотр в браузере | `LIVE_PREVIEW_INTERVAL_MS`, `LIVE_MAX_FRAME_WIDTH` |
| ANPR | Распознавание номеров | `ANPR_MAX_FRAME_WIDTH`, `ANPR_MIN_INTERVAL_MS` |

**Как экономится CPU без потери качества:**

1. **Детектор движения** — нейросеть только при движении в кадре (машина).
2. **Высокое разрешение для ANPR** — до 1920px + CLAHE для ночи.
3. **Низкое разрешение для live** — 960px, ~3 кадра/сек.
4. **Голосование** — в БД пишется номер, подтверждённый 2+ раза из 5 кадров.
5. **Общий lock** — камеры не грузят CPU одновременно.

**ROI** (зона дороги, доли 0–1): `CAMERA_1_ROI=0.1,0.3,0.8,0.5`

## Troubleshooting

**Камера offline** — проверьте RTSP URL с сервера:
```bash
ffprobe rtsp://user:pass@ip:554/stream1
```

**Медленное распознавание** — увеличьте `ANPR_MIN_INTERVAL_MS` или уменьшите `ANPR_MAX_FRAME_WIDTH`.

**Пропускает машины** — уменьшите `MOTION_MIN_AREA_RATIO` (например `0.002`) или увеличьте `MOTION_TAIL_SEC`.

**Ложные номера** — увеличьте `PLATE_VOTE_REQUIRED=3` и `MIN_CONFIRMED_CONFIDENCE=0.6`.

**Worker падает при старте** — первый запуск nomeroff_net скачивает модели (~2 GB). Убедитесь в наличии RAM и интернета.

**Неверный логин** — проверьте `ADMIN_USERNAME` / `ADMIN_PASSWORD` в `.env`, пересоздайте контейнер api.

## Структура данных

- `detections` — каждое срабатывание (номер, камера, фото, время)
- `vehicle_sessions` — сессии: въезд, выезд, статус (`on_site` / `completed` / `unknown`)
